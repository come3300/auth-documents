## Gitとは

![Photo by Minh Pham]( /images/documents/github-basic/github-logo.jpeg)

gitと一口で行ってもこの場合2つの意味合いがあるため
分けて解説します。

#### gitとは

<blockquote>
  バージョン管理システムの一つで、プログラムのソースコードやプロジェクトファイルの
  変更履歴を管理し、複数人での協力作業を支援するツールです
</blockquote>

#### githubとは

<blockquote>
  ソフトウェア開発者やプロジェクトチームがコードを共有し、
  協力して作業するためのプラットフォームです
</blockquote>

## Gitの目的

今回は Git / GitHub についての講座です。このGit/GitHub講座で全てを理解する必要はありません。
今後Git/GitHubの細かいコマンドは別途紹介します。まずはここで紹介する最低限のコマンドを
一種のルーティーンだと思って、レクチャー毎に実行して少しずつ慣れていきましょう。
Git とは、バージョン管理システムです。聞き慣れない言葉だとは思いますが、
使う目的は主に２つです。

① いつでもプログラムの更新を記録しておける

![Photo by Minh Pham]( /images/documents/tutorial/save-data.png)

開発をしていく中でプログラムは日々アップデートしていきます。
RPGなどのゲームをやっているときも、ゲームをやり終えるときにセーブをするのと一緒で、
プログラムを書いていく中でも、Gitを使い、コミット(更新を保存)をすることによって途中までの記録を残すことができます。
また、一昨日の記録に戻りたいとなったときには、その記録に戻ることも可能です。
そうすることによって、大きなミスをした時でもやり直しができるということです！
ラスボスを相手している時でも安心して挑戦できるわけです。

② チーム開発でのトラブルを防ぐ

![Photo by Minh Pham]( /images/documents/tutorial/git-team-trable.png)

Gitを使う目的はもう一つあります。それはチーム開発の時です。
例えば、二人同時に作業していたとして、同時に編集したとしましょう。
そうしたときに前に変更した内容が消えてしまうことがあります。
これは人と人とのコミュニケーションをどれだけ強化しても防ぎづらいことですし、
これを気にしながら作業するのも作業効率が落ちてしまいます。
そこでGitを使い共通のファイルのバージョンを管理することによって
それぞれのローカルのパソコンに独立した環境を作ることができ、
またその共通ファイルの更新を簡単にできます。
そうすることによって、お互いを干渉することなく開発を進めていくことができます。
そうすることによって、チーム開発でのストレスやトラブルを防ぐことができます。
そしてバージョン管理は現代の開発において当たり前に使われています。
なので学習期間で慣れて現場に出たときには気軽に触れるようにしましょう。

## 基本的な概念

Gitの理解を深めるために必要になってくるのがgitの概念を学習することです。

#### リポジトリ

![Photo by Minh Pham](/images/documents/tutorial/repository.png)
リポジトリとはファイルの変更履歴を保存しておく倉庫のような場所です。
実際には『.git』という場所にこの変更履歴が保存されています。

例えば
ファイルAにこんにちはという単語を追加した場合、
このこんにちわという単語が追加されたという記録が『.git』に保存されます。

#### ブランチ

![Photo by Minh Pham]( /images/documents/github-basic/ブランチの説明.png)

<Note>
ブランチとは、開発の作業用コピーを作る機能です。

**日常生活の例え：作文の下書き**

作文を書くときに、いきなり清書するのではなく、下書き用紙に書いてから清書しますよね。
ブランチはこの「下書き用紙」のようなものです。

- メインの作文（mainブランチ）を残したまま
- 別の紙（新しいブランチ）に色々試せる
- 上手くいったら清書に反映、失敗したら捨てられる

複数人で開発するときも、それぞれが自分の下書き用紙（ブランチ）を持てるので、
お互いの作業が邪魔にならないというわけです。
</Note>

#### コミット

![Photo by Minh Pham]( /images/documents/github-basic/git-commit-figure.png)

<Note>
コミットとは、変更内容を記録する「セーブポイント」のことです。

**日常生活の例え：RPGゲームのセーブ**

RPGゲームでボスに挑戦する前に、セーブポイントでセーブしますよね。
コミットはまさにこの「セーブ」と同じです。

- ゲームの進行状況を記録する = コードの変更を記録する
- 失敗したらセーブポイントからやり直せる = 前のコミットに戻れる
- セーブするタイミングは自分で決められる = コミットするタイミングも自分で決める

例えば、「ログイン機能を追加」というコミットを作れば、
後で「あのログイン機能を追加した時点に戻りたい」ということが簡単にできます。
</Note>

#### プッシュ

![Photo by Minh Pham]( /images/documents/github-basic/git-push-figure.png)

<Note>
プッシュとは、ローカルの変更をリモートリポジトリに「アップロード」することです。

**日常生活の例え：クラウドストレージへの保存**

GoogleドライブやiCloudに写真をアップロードすることを想像してください。
プッシュはこれと同じです。

- スマホ内の写真（ローカルリポジトリ） → クラウド（リモートリポジトリ）
- アップロードすることで：
  - 他の人と共有できる
  - 他のデバイスからもアクセスできる
  - バックアップになる

コミットだけではあなたのPC内に保存されているだけですが、
プッシュすることで初めてチームメンバーがあなたの変更を見られるようになります。
</Note>

## グローバルリポジトリへプッシュしてみよう

では新しくリポジトリを制作をして、そのリポジトリへ
プッシュしてみましょう！

### 1,新しくリポジトリを作る

ではまずグローバルの方に新しくリポジトリを作ります。
今回はtestというリポジトリを作成します。

### 2,ローカルにブランチを作る

作成したい階層で

```bash
mkdir test
//testという名のディレクトリが作成される
```

※testというディレクトリが作成されます。
ファイルの作り方がよくわからない方は
Downloadsに作成することをおすすめします。

```bash
cd test
//testディレクトリの階層へ移動

git init
//このコマンドを入力することで.gitファイルが作成される

ls -a
//.git というファイルが確認できるはずです。

touch readme.md
//readme.mdというファイルが作成されます。
ここにはそのリポジトリの説明が主に記されます

```

ここまでのコマンドを入力することでtestディレクトリに
readme.mdと.gitという隠しファイルが作成されていたら
次のステップへ行きましょう！

#### 3,addとcommitをしよう!

では次にご自身のPCに作った
ディレクトリとファイルを
ローカルにadd/commitをしていきます。

<InfoBox>
**なぜaddとcommitの2段階が必要なの？**

「いきなりcommitで保存すればいいのでは？」と思うかもしれませんが、
2段階に分かれていることには重要な理由があります。

**例：買い物カゴの仕組み**

ネットショッピングを想像してください。
- 商品を選ぶ（add） → カートに入れる
- レジに進む（commit） → 購入を確定する

カートに入れただけでは購入されません。レジで確定して初めて購入が完了します。

**Gitでも同じ：**
- `git add`：「この変更をセーブ対象にする」（カートに入れる）
- `git commit`：「セーブを確定する」（購入を確定する）

**これの何が便利？**
例えば、5つのファイルを変更したけど、そのうち3つだけをコミットしたい場合：
```bash
git add fileA.txt fileB.txt fileC.txt  # 3つだけ選択
git commit -m "A, B, Cの機能を追加"     # 選択したものだけコミット
```

もしaddがなかったら、変更した全ファイルが強制的にコミットされてしまい、
細かく履歴を管理することができません。
</InfoBox>

まずは下記コマンドを打ってみましょう。

```bash
git add .
```

このgit add .の『.』は全てのディレクトリとファイルをステージングへ追加する
時に使用します。

ディレクトリやファイルを追加したいときは

```bash
git add ファイル/ディレクトリ名
```

で追加できます。

次に下記コマンドを打ってみてください。

```bash
git commit -m "first commit"
```

この`git commit -m "first commit"`は

git commitと入力することでgit addでステージングされたファイルやディレクトリが
コミットされます。

そしてこの`-m "first commit"`の部分にはコメントを追加することができます。

具体的にどういう役割があるかというと
下記のようなリポジトリがあったとして

このリポジトリの下記の部分にコメントで入力した文字が反映されます。

こうすることでどういった作業を前に行ったかがわかりやすくなるという
メリットがあります。

#### 4,ローカルで作ったブランチをpushしてみよう

まずは下記コマンドを打ってみましょう。

```bash
git branch -M main
```

このコマンドを入力することでローカルに
`main`という名のブランチを作成することができます。

`-M`コマンドがメインブランチを登録するためのコマンドです。

そしてgit branch -M mainのmainがブランチ名になります
つまりメインブランチをmainという名前で作成したことになります!

ではメインとなるブランチの作成もできたので

```bash
git remote add origin git@github.com:ユーザー名/リポジトリ名.git
```

と入力します。

下記画像の部分に通常書いてあるので
それをそのままコピペすれば問題ありません。

```bash
git remote -v
```

と入力してみてユーザー名/リポジトリ名.gitがremoteとlocalで
同じであればしっかりと反映されています。

※このgit remote -vは行わなくても問題ありませんが確認したい方は
ぜひ入力してみてください。

最後に下記コマンドを入力します。

```bash
git push -u origin main
```

erorrも出ずに成功したら、無事グローバルの方に
あなたが作ったディレクトリをpushできています。

ここまでの流れができればgitの基礎的な
作業はクリアしています。お疲れ様でした。

### 豆知識 git addとcommitの違い

ワークツリーとは、実際に作業している場所(ディレクトリ)のことを指します。
ここでああでもあい、こうでもないと試行錯誤して開発を進めるわけですね。

そのあとソースコードが完成したら、リポジトリに更新する必要があります。
このソースコードの更新作業を「commit」と呼びます。

ここでワークツリーとリポジトリの間には、インデックスというスペースがあります。これはコミットするファイルを準備するところです。
ワークツリーからインデックスに登録することを「add」と呼びます。

※ちなみに、なんでインデックスが必要なの？いきなりワークツリーからリポジトリにコミットで良くない？と思った方もおられるかもしれません。

インデックスの役割としては、変更点をいきなりコミットすると更新されてほしくない箇所も更新する恐れがあるからです。色々変更を加えたけど、
ファイルAとBを変更したとき、Aだけコミットしたいことがあります。
インデックスにAだけを登録することで、変更して欲しいファイルだけコミットすることができるんですね。

コミットに含めたい変更を選別する作業を行うのがインデックスの役割です。

これで、git add とgit commitの違いがわかったかと思います。このあとpushコマンドを使って、
ローカルリポジトリの内容をリモートリポジトリに送信（アップロード）する流れになります。
